# OS-Lang Language Guide: Kernel & Systems Programming Superpowers

Welcome to the **OS-Lang Language Guide**. This document details the powerful low-level, memory-safe, and highly-optimized compiler features specifically designed for writing bare-metal operating systems, kernels, and embedded drivers.

By using OS-Lang, you get the performance of C and assembly with the developer ergonomics of modern languages.

---

## 1. Pointers and Memory Safety (`@unsafe`)

OS-Lang allows direct hardware manipulation and pointer arithmetic. To prevent accidental memory corruption in high-level business logic, OS-Lang enforces strict boundary checks: all direct memory access via pointers **must** reside within an `@unsafe` block.

### Declaring Pointers
You can declare pointers using the `*mut T` (mutable) or `*const T` (constant) syntax. Note: as of OS-Lang V2, the internal representation often uses `ptr` but the `*` prefix is fully supported in semantic types.

```os-lang
fn main() -> int:
    let mut address: int = 0xB8000
    let mut vga_buffer: ptr = address as ptr
    
    # Safely guarded raw memory access
    @unsafe:
        let val: int = *vga_buffer
        *vga_buffer = 0x0F41  # Write 'A' to the VGA text buffer
        
    return 0
```

## 2. Hardware Alignment: `@packed` and `sizeof`

When mapping structs directly to hardware memory (e.g., GDT entries, IDT descriptors, VGA tables), LLVM padding can corrupt the memory layout. 

Use the `@packed` decorator to enforce zero-padding on a struct. You can also use the built-in `sizeof()` intrinsic to programmatically calculate byte widths at compile time.

```os-lang
@packed
struct IdtEntry {
    offset_low: u16
    selector: u16
    ist: u8
    type_attr: u8
    offset_mid: u16
    offset_high: u32
    zero: u32
}

fn initialize_idt() -> void:
    # sizeof calculates byte-width at compile time
    let entry_size: int = sizeof(IdtEntry)
    print(entry_size) # Outputs exactly 16 bytes, thanks to @packed!
```

## 3. Enums and Pattern Matching (`match`)

Instead of managing floating integer constants for states, OS-Lang provides zero-cost `enum` types and the `match` statement. Under the hood, `match` statements are compiled down directly to highly optimized LLVM `switch` tables, ensuring `O(1)` branching.

```os-lang
enum ThreadState {
    RUNNING,
    BLOCKED,
    SLEEPING,
    ZOMBIE
}

fn process_thread(state: ThreadState) -> void:
    match state:
        ThreadState.RUNNING =>:
            print("Executing...")
        ThreadState.BLOCKED =>:
            print("Waiting for I/O")
        _ =>:
            # Wildcard default catch
            print("Inactive Thread")
```

## 4. Hardware I/O Intrinsics

You do not need to write raw assembly for simple CPU operations or port-mapped I/O. OS-Lang includes a suite of hardware intrinsics that compile down to single `x86_64` CPU instructions.

- `halt()` -> `hlt`
- `cli()` -> `cli`
- `sti()` -> `sti`
- `rdtsc()` -> `rdtsc`
- `inb(port)`, `outb(port, val)` -> Port Mapped I/O

```os-lang
fn reboot_system() -> void:
    cli()  # Clear interrupts
    let keyboard_port: int = 0x64
    let reset_command: int = 0xFE
    
    # Send reset command to PS/2 controller
    outb(keyboard_port, reset_command)
    halt() # Halt CPU if reset fails
```

## 5. Hardware Interrupts (`@interrupt`)

Writing interrupt handlers in C typically requires inline assembly wrappers to push/pop registers and handle the `iretq` instruction.

In OS-Lang, you just use the `@interrupt` decorator. The compiler automatically assigns the `x86_intrcc` calling convention, saving the CPU state correctly and generating a safe `iretq` return.

```os-lang
@interrupt(32)
fn timer_handler() -> void:
    # Acknowledging PIC End of Interrupt (EOI)
    outb(0x20, 0x20)
    print("Timer Tick!")
```
